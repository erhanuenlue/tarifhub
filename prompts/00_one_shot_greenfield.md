# 00 — The one-shot greenfield prompt (reference)

> How the orchestrator would prompt itself to build TarifHub from an empty directory in one autonomous session. Not the CAS route (that's 01–07, because graded evidence needs contemporaneous human curation) — this is the reference for outcome-prompting a whole system: lock the invariant and the product shape, define done as falsifiable evidence, grant autonomy with ADRs as after-the-fact accountability. Steps, stack and sequencing stay with the model.

---

You're starting from an empty directory. Build TarifHub — I'll give you the idea, the invariants, and what "done" means. Everything else is yours: plan it, decide it, build it, verify it. Decide alone and record decisions as you go; surface only what is irreversible, legally risky, or costs real money.

**The idea.** Switzerland replaced its outpatient medical tariff on 1 Jan 2026 (TARMED → TARDOC + flat rates) and now revises it annually. The billing data lives in ~110 tariff types across 20+ public sources — XLSX, XML, PDF, FHIR R5 — on independent update cycles, with no machine-readable source of truth, so every software vendor rebuilds the same brittle pipeline. The product: harvest the public sources, harmonise them into one canonical, versioned record model — AI does the mapping, humans approve uncertain cases — then **freeze** each approved record (content hash, version, full lineage) and serve frozen records through a clean API: REST + FHIR read, point-in-time and diff queries, semantic search, and an MCP server so AI agents can consume it. On top, a small web console: master-detail search over frozen records, plus the review queue where a human approves or corrects AI mapping proposals before freeze.

**The invariant that is the product — never negotiable:** no AI computes, mutates, or invents an authoritative billing value at serve time. AI works only before the freeze (field mapping, confidence scoring, anomaly flagging — non-billing fields only) and in clearly-labelled explain/search seams that never alter values. Same query → same answer, forever, with provable provenance. Build the **enforcement**, not the promise: a static test that fails CI if the value-serving path can import an LLM client; immutable rows (an update is a new version); an append-only audit log; idempotent re-ingestion by content hash. In the console, frozen values and AI prose stay visually and architecturally separate — frozen values in navy + monospace with version and hash visible; AI output always on an explicitly labelled surface ("AI-generated — not a billing value").

**Constraints.** Public (BAG/opendata.swiss) sources only. Start with two of maximal format diversity — one flat XLSX (the EAL lab list), one hierarchical FHIR R5 (the new Spezialitätenliste) — real files, committed as trimmed fixtures. One person maintains this: one language end-to-end where possible, boring proven components, few moving parts. No agent frameworks in the data path, no separate vector DB, no Kafka, no service mesh — if you believe you need one, write the ADR that justifies it and then don't build it. German designations canonical, FR/IT optional. Money as decimals, never floats. Tests run fully offline — no network, no API key; the AI seam needs a deterministic fallback. Containerise each sub-system separately; compose for local; CI that lints, tests, runs the boundary test visibly, and scans for secrets and vulns.

**Done means all of this is true, with evidence shown:** (1) `docker compose up` plus one command ingests both real sources end-to-end — parse → AI-map (live with a key, fallback without) → validate → score → review queue → freeze — and re-running on identical inputs creates zero new versions; show the hash-identity test output. (2) The API serves a frozen record by code, at a point in time, as a diff between two versions, via semantic search, and through MCP tools — each with a test. (3) The console renders master-detail and the review form against the live API; approving a queued record freezes it server-side. (4) The boundary test genuinely guards the import graph — prove it by making it fail on a deliberate violation, then revert. (5) A stranger goes from clone to running system in under ten minutes using only the README — verify by following your own README literally in a fresh directory. (6) `docs/` contains the architecture in prose and diagrams, every non-obvious decision as a short ADR, and an honest `LIMITS.md`: what you cut, what's weak, what comes next.

**Working style.** Plan first, then go. When something fails, show the real output, fix it, note what caught it. Before declaring done, run a fresh-context review of the entire repo against this prompt — a hostile senior engineer who has never seen it must sign off — and fix what they would flag. Then report: what exists, what's proven (quoted test output), what's deliberately absent, and the three biggest risks in the codebase. The simplest thing that works well end-to-end beats the impressive thing that almost works.
