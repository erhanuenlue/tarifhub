[~15s] Guten Tag. Das ist tarifhub. Es nimmt unübersichtliche Schweizer Tarifdaten aus dem ambulanten Bereich und macht daraus etwas, dem man wirklich vertrauen kann. Ich bin Erhan. In fünf Minuten zeige ich Ihnen drei Dinge: die Idee, das Produkt, und vor allem, wie es mit einem KI-gestützten Auto-Loop entwickelt wurde.

The problem

Tariff data you cannot trust is worthless.

  • ~110 tariff types, 20+ sources
  • XLSX, XML, PDF, FHIR: no single machine interface
  • One uncaught mapping error becomes a wrong invoice
[~32s] Schweizer Tarifdaten sind zersplittert: rund hundertzehn Tarifarten, über zwanzig Quellen, verteilt auf Excel, XML, PDF und FHIR, und keine einzige maschinenlesbare Schnittstelle. Sie kommen von verschiedenen Behörden, in drei Sprachen, und ändern sich ständig. Das eigentliche Problem: Ein einziger unbemerkter Mapping-Fehler wird am Ende zu einer falschen Rechnung. Ein Abrechnungswert muss exakt, nachvollziehbar und reproduzierbar sein. Vertrauen ist hier kein Feature, sondern das ganze Produkt.

The vision

One versioned, deterministic API.

  • Four layers, split by one freeze line
  • Read surface: REST · FHIR R4 · MCP
  • Every record carries its proof: a SHA-256 hash

above · AI-assisted, pre-freeze

L0 harmonise  adapters → ai_map → validate → score
FREEZE LINE immutable · hashed · versioned
L1 serving  TarifCore (REST + FHIR R4) · TarifMCP
L2 rules  TarifIQ (deterministic)
L3 apps  TarifGuard console

below · fully deterministic, read-only

[~33s] Die Antwort ist eine einzige, versionierte und deterministische API. Vier Schichten, getrennt durch eine Linie, die Freeze-Linie. Oberhalb wird harmonisiert, KI-gestützt, vor dem Einfrieren; unterhalb wird ausgeliefert, vollständig deterministisch, über REST, FHIR R4 und ein MCP für KI-Agenten. Vier Sub-Produkte sitzen darauf: TarifCore liefert aus, TarifMCP bedient Agenten, TarifIQ kapselt die Regeln, die TarifGuard-Konsole macht alles sichtbar. Und jeder Datensatz trägt seinen Beweis: einen SHA-256-Hash.

The one rule

Harmonise above. Freeze. Serve below.

FREEZE LINE SHA-256 · append-only audit
[~37s] Das ist das Herzstück. Verfolgen wir einen Datensatz. Oberhalb der Linie wird er harmonisiert: Hier, und nur hier, darf die KI arbeiten, vor dem Einfrieren und nur auf nicht-abrechnungsrelevanten Feldern. An der Linie wird er eingefroren: ein SHA-256-Hash über den kanonischen Inhalt, jede Änderung wird eine neue Version, das Audit-Log ist nur anhängbar. Unterhalb wird er ausgeliefert, schreibgeschützt und deterministisch, über REST, FHIR und MCP. Die eine Regel, die alles definiert: Keine KI berechnet oder verändert jemals einen Abrechnungswert zur Auslieferungszeit.

The development journey

An empty repo and a definition of done.

  • Not a backlog: a falsifiable "done"
  • Outcome prompts, not step-by-step scripts
  • One human gate I keep for myself: I approve the plan
[~25s] Jetzt zum eigentlichen Beitrag: wie es gebaut wurde. Ich habe nicht mit einer Aufgabenliste angefangen, sondern mit einem leeren Repository und einer überprüfbaren Definition von fertig: Prompts, die das Ziel und die Verifikation beschreiben, aber bewusst nicht die Schritte. Moderne Modelle arbeiten besser mit einem falsifizierbaren Ziel als mit einem Skript. Das Einzige, was ich mir immer vorbehalte: Ich gebe den Plan frei.

How it was built

/ship: a multi-model assembly line.

  • Orchestrator plans, workers build test-first
  • An independent model family reviews every PR
  • Nine phases, one human gate: the plan
[~33s] Gebaut wird über die /ship-Pipeline, neun Phasen von der Planfreigabe bis zum Merge. Aufgeteilt wird dort, wo Qualität sich auszahlt: Ein Orchestrator, Claude Opus 4.8, plant und beurteilt; Worker-Agenten implementieren test-first und sammeln Belege. Dann prüft eine zweite, unabhängige Modellfamilie, OpenAI gpt-5.5, jeden Pull Request, weil ein Claude-geprüftes Claude-Diff dieselben blinden Flecken teilt. Siebenundzwanzig Pull Requests sind so gemerged worden, jeder von dieser zweiten Familie geprüft.

A process that outlives its model

Built across a model transition.

  • Early blocks: Fable 5  →  switch  →  Opus 4.8
  • Pinned in one place; workers always Opus 4.8
  • Gates, board, CI are model-agnostic
[~22s] Ein Detail, das mir wichtig ist: der Modellwechsel. Die frühen Blöcke liefen auf Claude Fable 5; dann bin ich auf Claude Opus 4.8 gewechselt, das stärkste mir verfügbare Modell, festgelegt an genau einer Stelle. Die Worker liefen durchgehend auf Opus. Dass der Prozess diesen Wechsel ohne Anpassung überstanden hat, ist selbst ein Beleg: Gates, Board und CI sind modell-unabhängig.

The headline

The machine writes its own next step.

  • Fixed loop: replays a known list
  • Self-prompting loop: optimises a moving target
  • Runs unattended; checkpoints its state
[~33s] Das ist die Schlagzeile. Es gibt zwei Schleifen. Die feste Schleife spielt eine bekannte Liste von Prompts ab, wie eine Aufnahme. Die selbst-promptende Schleife geht weiter: Jede Iteration liest den Zustand, schreibt sich die nächste Aufgabe selbst, lässt sie durch /ship laufen, prüft einen Vertrag und wiederholt das bis zu einem messbaren Ziel. Das funktioniert nur, weil fertig maschinell überprüfbar ist. Ohne diesen Gradienten würde das Selbst-Prompten einfach abdriften.

The intellectual core

Autonomy is safe because the boundary is enforced.

  • Fitness ratchet: 62/62 checks, never regress
  • Freeze-line guard plus a boundary test
  • Green-contract: merge only on full green
[~35s] Das ist der intellektuelle Kern. Vier Gates lassen die Schleife hart laufen und trotzdem sicher bleiben. Erstens: eine deterministische Fitnessfunktion, heute bei zweiundsechzig von zweiundsechzig Elementen, mit einer Basislinie, die nur nach oben ratscht. Zweitens: die Freeze-Linie, geschützt durch einen Hook und einen Test, der sicherstellt, dass auf dem Wertpfad kein Modell-Client importierbar ist. Drittens: der Green-Contract regelt den autonomen Merge. Viertens: ein Secret-Gate bei jeder Iteration. Die KI darf überall sonst hart laufen, genau weil diese Linie nicht still überschritten werden kann.

Quality and CI/CD

Every change passes the gates a reviewer would.

  • A second model family reviews every PR
  • Full CI: lint, tests, boundary, secrets, SBOM
  • Centralised RFC-7807 error handling (ADR-019)
  • Docs and diagrams generated by the loop
[~25s] Die CI prüft bei jedem Push dieselben Gates wie ein Reviewer: Linting, die komplette Testsuite, die Determinismus-Grenztests, Secret- und SBOM-Scans. Fehler beantwortet die Serving-API zentral im RFC-7807-Format (ADR-019). Die Zahlen sind konkret: Der e5-Query-Prefix-Fix hob die mehrsprachige Trefferquote von null Komma acht drei drei auf null Komma neun eins sieben. Und jedes Diagramm hier wurde von der Pipeline erzeugt und als statische Datei eingecheckt.

The result

What actually shipped.

  • Built: BAG EAL (XLSX) + BAG ePL/SL (FHIR R5)
  • Serving · MCP · rules · console, on 319 green tests
  • Vision, not built: TARMED / TARDOC

TarifGuard console: search → frozen record → labelled AI explain panel.

[~25s] Was ist tatsächlich entstanden? Zwei echte Quellen sind angebunden: die Analysenliste als Excel und die Spezialitätenliste als FHIR R5. Darauf laufen die Serving-API, das MCP, die Regel-Schicht und die Konsole, abgesichert durch dreihundertneunzehn grüne Tests. TARMED und TARDOC bleiben bewusst Vision, nicht gebaut. Ehrlicher Umfang schlägt vorgetäuschte Vollständigkeit, und ist im Bericht auch so markiert.

What I learned

An enforced, tested definition of done is what makes AI autonomy trustworthy.

  • Give the system a gradient it can measure
  • And boundaries it cannot cross
  • The method is now a reusable framework
[~20s] Das Nützlichste ist nicht dieses Projekt, sondern die Methode, verpackt als wiederverwendbares Framework. Die Lektion verallgemeinert sich: Gib dem System einen Gradienten, den es messen kann, und Grenzen, die es nicht überschreiten kann, dann trägt es die Arbeit. Und die eine Grenze, die ich nie an die KI delegiert habe: kein Abrechnungswert zur Auslieferungszeit.
