[~15s] Good morning. This is TarifHub: it takes messy Swiss outpatient tariff data and turns it into something you can actually trust. I am Erhan. In five minutes I want to show you the idea, and more importantly, how it was engineered with an auto-loop.

The problem

Tariff data you cannot trust is worthless.

  • Many sources, four languages, constant new versions
  • A billing value must be exact, traceable, reproducible
  • Trust, meaning determinism, is the product
3 sources·4 languages ·N versions →1 certified value
[~30s] Tariffs arrive from several authorities, in German, French and Italian, and they change constantly. A billing value has to be exact and reproducible: the same input must always give the same output, and you must be able to prove it later. So trust is not a feature here, it is the whole product. One cautionary moment: an AI scaffold once used the wrong field contract and rendered every value as a dash. Reading both sides of the wire caught it, not a test. Determinism is the point.

The idea

Four layers.
One freeze line.

  • L0 harmonise: AI-assisted, pre-freeze
  • Freeze line: records become immutable, hashed
  • L1 to L3 serving: fully deterministic
[~35s] The architecture is four layers split by one line, the freeze line. Above it, layer zero harmonises raw data, and that is the only place AI is allowed to run, before freezing, on non-billing fields. At the line, every record is frozen: a SHA-256 hash over its canonical content, updates are new versions, the audit log is append-only. Below the line, serving is deterministic, read-only. The one rule that defines the whole system: no AI ever computes or mutates a billing value at serve time. The freeze line is the trust guarantee.

How I started

An empty repo and a definition of done.

  • Not a backlog: a falsifiable "done"
  • Outcome prompts, not step-by-step scripts
  • One human gate: I approve the plan
[~25s] I did not start with a task list. I started with two checked-in control files, the project facts and the workflow, plus outcome prompts. Each prompt states the goal, the constraints, and the verification that proves done, and deliberately leaves out the steps. Modern models do better with a falsifiable target than with a script written for an older model. The only thing I always keep for myself is approving the plan.

How it was built

/ship: a multi-model assembly line.

  • Orchestrator plans, workers build test-first
  • An independent model reviews every PR
  • Nine phases, one human gate: the plan
[~35s] The build runs through a pipeline called ship, nine phases from plan approval to merge. The work is split by where quality compounds. An orchestrator, Claude Opus 4.8, plans and judges; worker agents implement test-first and collect runtime evidence. Then a second, independent model family, OpenAI gpt-5.5, reviews every single pull request, because a Claude-reviewed Claude diff shares blind spots. Twenty pull requests were merged this way, each one reviewed by that second family.

The headline

The machine writes its own next step.

  • Fixed loop: replays a known list
  • Self-prompting loop: optimises a moving target
  • Runs unattended; checkpoints its state
[~40s] This is the headline of the project. There are two loops. The fixed loop runs a known list of prompts back to back, like replaying a recording. The self-prompting loop goes further: each iteration reads the current state, writes the next task as a prompt by itself, runs it through ship, checks a contract, and repeats until a measurable goal is reached. The difference matters: the fixed list replays, the self-prompting loop optimises whatever is most valuable now. It only works because done is machine-checkable. Without that gradient, self-prompting just drifts.

The intellectual core

Autonomy is safe because the boundary is enforced.

  • Fitness ratchet: ~60 checks, never regress
  • Freeze-line guard plus a boundary test
  • Green-contract: merge only on full green
[~45s] This is the intellectual core. Four gates let the loop run hard and still stay safe. One: a deterministic fitness function expresses done as about sixty machine-checkable elements, with a committed baseline that only ratchets up, today at 63 of 63. Once an element passes it must keep passing, or CI fails. Two: the freeze line is guarded by a pre-tool hook that blocks any edit below it, plus a test that asserts no model client is even importable on the value path. Three: the green-contract governs autonomous merge, CI green, findings dispositioned, no unauthorised frozen-path change, clean tree, anything less stops for me. Four: a secret gate on every iteration. The agent can run hard everywhere else precisely because this line cannot be crossed silently.

Quality and CI/CD

Every change passes the gates a reviewer would.

  • A second model family reviews every PR
  • Full CI: lint, tests, boundary, secrets, SBOM
  • Docs and diagrams generated by the loop
[~25s] CI runs the same gates a reviewer would, on every push: lint, the full test suite, the determinism boundary tests, secrets and SBOM scans. The numbers are concrete: the e5 query-prefix fix lifted cross-lingual recall at five from 0.833 to 0.917. Every diagram here was generated by the pipeline and committed as a static file.

Observable end to end

The whole system is inspectable while it runs.

Shipboard: the live /ship pipeline, per-agent cost, and the CAS grading floor. Captured while this deck was being built.

[~20s] One dependency-free dashboard makes the system legible: it reads real pipeline events, git state, and the fitness function. Its rule is that it never asserts a state it cannot show evidence for; it says unknown rather than invent progress. That is what lets me trust it while the loop runs unattended.

What I learned

An enforced, tested definition of done is what makes AI autonomy trustworthy.

  • Give the system a gradient it can measure
  • And boundaries it cannot cross
  • The method is now a reusable framework
[~20s] The most useful outcome is not this project, it is the method, packaged as a reusable framework: a bootstrap prompt that scaffolds a new project with its gates, dashboard, and loop. The lesson generalises: give the system a gradient it can measure and boundaries it cannot cross, and it carries the work.
